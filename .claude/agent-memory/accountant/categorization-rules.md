# Карта категорий — метка операции Bangkok Bank → категория

Знак берём из дельты баланса (см. [bbl-statement-parsing.md](bbl-statement-parsing.md)): `amount<0` = расход, `amount>0` = доход.

## Расходы (amount < 0)
| Метка (Particulars) | Категория |
|---|---|
| `PMT FOR GOODS`, `PURCHASE E-CHN` | 🍜 Еда и магазины (оплата товаров/карта) |
| `TRF. PROMPTPAY`, `PMT. PROMPTPAY`, `TRANSFER` (исходящий), `TRF TO OTH BK` | 💸 Переводы |
| `CASH W/D ATM`, `CASH ATM W/D` | 🏧 Наличные (ATM) |
| `COM/ANNUAL FEE`, `FEE OTH BAK ATM` | 🧾 Комиссии |

## Доход (amount > 0)
| Метка | Категория |
|---|---|
| `TRF FR OTH BK`, `TRANSFER` (входящий/возврат) | 📥 Поступления |
| `BE1ST SMART` | ♻️ Кешбэк / возврат (постится задним числом) |

## Спец-правило: Аренда / Дом
Повторяющийся **фиксированный** перевод PromptPay у начала месяца (около 1-го числа; может пройти 31-м числом предыдущего). Выделяется из «Переводов» в отдельную категорию 🏠 Дом.
- **Сумму НЕ хардкодить — подтверждать у пользователя.**
- Детектор: `PROMPTPAY` с суммой == подтверждённая аренда. Проверить, что нашлось по платежу на каждый месяц; если в каком-то месяце нет — сказать пользователю (аренда могла пройти иначе, напр. через `PMT FOR GOODS`).

## Важное ограничение: у переводов НЕТ имён получателей
В выписке Bangkok Bank строки переводов содержат только тип операции (`PROMPTPAY` и т.п.), без имени/телефона получателя. Поэтому переводы конкретному человеку (напр. «Мария») **автоматически не выделить** — это самый крупный и «слепой» котёл расходов.
Решение: интерактивная пометка в дашборде (пользователь кликает операции → тег → отдельная категория и сумма пересчитываются). Не приписывать получателя наугад.

**Уточнено пользователем (правило для этого аккаунта):** исходящие `TRANSFER` (via mPhone) = переводы конкретному человеку («Мария»). Выделять в отдельную категорию 👤 (цвет `#e06fa0`), вынимая из общих «Переводов». Берётся ВЕСЬ исходящий `TRANSFER` (не только круглые суммы), хотя в хвосте бывают редкие служебные обнуления баланса с некруглыми суммами — пользователь решил их не отсеивать. `PROMPTPAY` к Марии не относится. По запросу показывать помесячный объём (бар по месяцам).

## Детский сад / школа (паттерн «крупный платёж + еженедельная еда из каши»)
Крупный разовый PromptPay-платёж (оплата/семестр) пользователь выделяет в отдельную категорию (напр. 🎒 «Детский сад»). Связанная **регулярная еда** (по выходным) тонет в общей «каше» переводов: суммы плавают (≈800–1000), уходят PromptPay/переводом разным людям, без меток — точную не вытащить.

Итоговый подход (пользователь предпочёл реальные деньги, а НЕ синтетический бюджет, и БЕЗ отдельной секции/карточки):
- крупный платёж → в категорию по сумме (реально);
- еду → **переносить по одной реальной PromptPay-транзакции 780–1000 ฿ за каждую ISO-неделю после даты крупного платежа, рандомно (seeded random для воспроизводимости)**, ТОЛЬКО из категории `transfers` (PromptPay/третьи лица). **Марию (TRANSFER) не трогать** — выходная еда уходит третьим лицам, а не Марии. Реассайн `cat → kinder`. Это re-bucketing: сверка с банком сохраняется, «Переводы» уменьшаются. Садик живёт как обычная категория (донат/легенда/график/таблица) — отдельная карточка не нужна.
  (История: сначала пробовали синтетический бюджет 1000/нед — отвергнут; потом тянули ближайшее к 1000 из maria+transfers — тоже отвергнуто, т.к. Марию трогать нельзя.)

Конкретные суммы/даты не хранить — берутся из выписки и подтверждаются пользователем.

## Заметки
- `TRANSFER` бывает и расходом, и доходом — разводить по знаку дельты, не по метке.
- `BE1ST SMART` — всегда зачисление (кешбэк/возврат округлений), идёт с датой задним числом.
