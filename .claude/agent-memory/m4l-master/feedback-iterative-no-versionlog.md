Во время итеративной работы над девайсом НЕ документировать детали каждой версии.

**Прекратить:**
- писать журнал/log-запись на каждую правку;
- обновлять wiki-страницы на каждый микро-шаг;
- вести подробный changelog по каждой версии (раздутый «Журнал правок» в памяти с историей всех билдов).

**Что остаётся ВСЕГДА:**
- дата-бэкап перед каждой правкой — это откат, обязателен без исключений (см. процедуру в [[controlxl-project-map]]);
- актуальное «текущее состояние» в project-map памяти: текущий md5, какие объекты есть, проводка — кратко, **перезаписывать актуальным, а не копить историю версий**.

**Подробная документация** (wiki-страница фичи, развёрнутый разбор) — ТОЛЬКО когда фича финализирована, либо по явной просьбе пользователя.

**Why:** фиксация деталей каждой версии при быстрой итерации = шум и трата времени; история раздувает память и устаревает за часы. Откат обеспечивает дата-бэкап, а не текстовый журнал. Правило введено пользователем 2026-06-06.

**How to apply:**
- В процедуре правки шаги = backup → правка → валидация → обновление КРАТКОГО current-state. Шаг «обнови wiki/log журнал версий» из процедуры убран.
- current-state в [[controlxl-project-map]] и [[xl-performance]] держать одним актуальным срезом (md5 + объекты + проводка), не списком билдов.
- Browser Load (на 2026-06-06) — ещё в активной отладке, НЕ финализирован: развёрнутую wiki по нему делать только по завершении. Уже написанную wiki задним числом НЕ чистить — просто впредь не плодить.
- Прежнее железное правило «Документируй фичи — сразу и честно / запись в wiki/log.md в тот же день» относится к ФИНАЛИЗИРОВАННЫМ фичам и к явной просьбе, а не к каждому итеративному билду.
