# Директива: Индексирование существующей дизайн-системы (ds_scan)

> **Если в `ds/` уже лежит демо-ДС из стартового набора (CryptoVPS):** ты идёшь со СВОИМ продуктом — собери `ds/` из своего Figma и **перезапиши** демо-файлы. Старое (CryptoVPS) не сохраняй и не смешивай. Отдельно просить агента «переделай» не нужно — запуск этой директивы на своём файле и есть замена.


## Задача
Превратить существующую ДС в Figma (свою, корпоративную или дублированную из Figma Community) в компактный индекс, который агент держит в контексте при сборке экранов. Читаются только Variables и верхний уровень страницы Components — без рекурсии в инстансы. На выходе — `ds/foundation.md` (токены) и `ds/components.md` (каталог компонентов с матрицами вариантов и Node ID).

## Роль
Дизайн-системный архитектор с опытом аудита больших ДС (300+ компонентов). Знает, как работают Figma Variables и `componentPropertyDefinitions`. Привычка читать индексы, а не дерево целиком.

> Адаптируй роль под источник:
> - Корпоративная ДС → аудитор, привыкший работать с unfamiliar ДС, не пытается переделать
> - Дублированная Community-библиотека → дизайнер, который понимает базу (Shadcn, Material, Carbon, Gravity UI) и знает, какие имена компонентов ожидать
> - Своя ДС, выросшая до 200+ компонентов → дизайн-системщик, помнит часть наизусть, проверяет описания

## Как устроена директива

Три фазы. Первая — пользователь даёт правильную ссылку (страница Components, не корень файла) и контекст. Вторая — `get_variables` собирает токены, агент пишет `ds/foundation.md`. Третья — обход верхнего уровня страницы Components, агент пишет `ds/components.md`. Финал — короткий self-check на полноту и пустые места.

Жёсткое правило: **никаких рекурсивных обходов**. Не заходим в инстансы, не открываем варианты внутри наборов, не ходим вглубь фреймов. Читаем индекс — и всё.

**Важно про инструмент:** `use_figma` (для `get_variables` и обхода children страницы Components). Не использовать read-tools, которые тянут весь файл.

**Важно про доступ:** для корпоративной ДС у пользователя должен быть просмотровый доступ к файлу. Для Community-библиотеки — сначала Duplicate to your drafts, потом сканирование уже своей копии.

## Фаза 1. Получение контекста

Один блок вопросов:

```
1. Источник ДС: своя / корпоративная / дублированная из Figma Community.
2. Ссылка строго на страницу «Components» в файле ДС (не на корень файла).
3. Если есть отдельный файл с фундаментом — ссылка на него (или пометка «фундамент в том же файле»).
4. Локаль продукта: ru / en / ru+en. Влияет на проверку шрифтов на кириллицу.
```

**Если ссылка ведёт на корень файла или другую страницу** — переспроси. Главная причина падений `ds_scan` — рекурсивное чтение всей ДС подряд при ссылке на корень. Стоп-фраза: «нужна ссылка именно на страницу Components, иначе MCP захлёбывается на больших ДС».

**Если источник — Community-библиотека и пользователь не делал Duplicate** — попроси сначала продублировать к себе и прислать ссылку на копию. Сканирование read-only Community-файла ограничено и теряет половину контекста.

Сохрани ответы в `ds/source.md`. Это якорь для всех решений по доводке.

## Фаза 2. Foundation — Variables

### Фаза 2.1: Чтение Variables

Один вызов `use_figma`:

```js
const allVariables = figma.variables.getLocalVariables();
const collections = figma.variables.getLocalVariableCollections();

// Группируем по коллекциям, для каждого variable забираем имя, тип, значение в default mode
// (или alias, если значение — VARIABLE_ALIAS на другую переменную)
```

Это плоский запрос. На ДС с тысячью токенов отрабатывает за секунды, в контексте занимает мало.

### Фаза 2.2: Сборка `ds/foundation.md`

Структура файла — по тем же двум-трём слоям, что и в `ds_baseline`. Если в ДС привязки между коллекциями уже сделаны — отображай как есть. Если все Variables в одной плоской коллекции (бывает в старых ДС) — пометь это и предложи раздельно.

```markdown
# Foundation (scan)

**Источник:** <строка из source.md>
**Дата сканирования:** <ISO-дата>

## Слой 1 — примитивные токены

### Палитра
- accent-500 = #2563EB
- accent-100, accent-300, accent-700, accent-900
- gray-00 ... gray-950
- success-500, warning-500, error-500, info-500

### Типографика
- font-sans = Inter
- size-xs = 12 ... size-4xl = 48
- weight-regular ... weight-bold
- line-height-base = 1.5

### Радиусы
- radius-none = 0, radius-sm = 4, radius-md = 8, radius-lg = 16

### Отступы
- space-1 = 4 ... space-16 = 64

## Слой 2 — смысловые токены
*только если в исходной ДС есть отдельная коллекция с привязками*

- surface-default → gray-00
- surface-action-primary → accent-500
- text-default → gray-900
- ...

## Слой 3 — компонентные токены
*если коллекция найдена — перечислить, иначе раздел опустить*

## Замечания после сканирования
- [ ] Привязки Semantic → Primitive: <«сделаны» / «частично» / «не сделаны, плоская коллекция»>
- [ ] Кириллица в шрифте: <«поддерживается» / «не проверена» / «отсутствует»>
- [ ] Дубликаты имён: <если есть>
- [ ] Variables, не использующиеся ни в одном компоненте: <если детектируется>
```

Покажи файл пользователю. Пометь места, где сканирование не уверено (например, цвет назван `c-1` — не понятно, это акцент или нейтрал — сразу попроси уточнить).

## Фаза 3. Components — каталог

### Фаза 3.1: Обход верхнего уровня

Один вызов `use_figma`. Берём страницу «Components» по ссылке из фазы 1 и читаем только её прямых детей.

```js
// page = страница Components по ссылке
const topLevel = page.children;
// topLevel — это набор компонентов (ComponentSetNode), отдельные компоненты (ComponentNode), фреймы-группировщики

const catalog = [];
for (const node of topLevel) {
  if (node.type === "COMPONENT_SET") {
    catalog.push({
      name: node.name,
      description: node.description,
      properties: node.componentPropertyDefinitions,  // матрица вариантов
      nodeId: node.id,
    });
  } else if (node.type === "COMPONENT") {
    catalog.push({
      name: node.name,
      description: node.description,
      properties: null,  // одиночный компонент без вариантов
      nodeId: node.id,
    });
  } else if (node.type === "FRAME" || node.type === "SECTION") {
    // фрейм-группировщик: записать как раздел и продолжить обход на 1 уровень глубже,
    // НО не заходить в инстансы
    for (const child of node.children) {
      if (child.type === "COMPONENT_SET" || child.type === "COMPONENT") {
        catalog.push({...});
      }
    }
  }
  // в инстансы (INSTANCE) и в children компонентов НЕ заходим
}
```

**Ключевое:** обход — на один уровень в глубину при наличии группирующих фреймов или секций, не дальше. Внутрь компонентов и инстансов не заходим. Это и есть основная защита от таймаута.

**Важно для Beefy-подобных ДС:** если компоненты разнесены по отдельным страницам (а не на единой странице Components) — сканируй все страницы параллельно (по одному `use_figma` на страницу в одном сообщении). Результаты сливаешь в один `ds/components.md`.

### Фаза 3.2: Сборка `ds/components.md`

Каждый компонент — три-четыре строки в фиксированном формате:

```markdown
# Components (scan)

**Источник:** <ссылка на страницу Components>
**Кол-во наборов компонентов:** <N>

## Действия

### Button
- **Назначение:** основная кнопка действий, primary / secondary / ghost
- **Варианты:** type: primary | secondary | ghost; size: sm | md | lg; state: default | hover | disabled | loading
- **Node ID:** `1:234`

### IconButton
- **Назначение:** квадратная кнопка под иконку, для тулбаров и компактных секций
- **Варианты:** size: sm | md | lg; state: default | hover | disabled
- **Node ID:** `1:256`

## Ввод

### Input
- **Назначение:** однострочное поле ввода с лейблом и подсказкой
- **Варианты:** state: default | focus | error | disabled; size: sm | md | lg
- **Node ID:** `1:301`

### Select
...

## Контейнеры

### Card
- **Назначение:** контейнер с заголовком, телом, опциональными действиями
- **Варианты:** padding: compact | default; elevation: flat | raised
- **Node ID:** `1:412`

## Навигация

### Navbar
...

## Фидбэк

### Toast
...

## Замечания после сканирования
- [ ] Компоненты без описания: <список имён, у которых description пустой>
- [ ] Старые версии рядом с новыми: <если детектируется по именам типа «Button old», «Button v1»>
- [ ] Компоненты, чьи варианты в matrix не объяснены: <если есть тип variant без понятных значений>
```

**Правила записи:**
- Группировка по разделам (Действия, Ввод, Контейнеры, Навигация, Фидбэк, Данные, Прочее) — по смыслу из имени или поля description. Если у компонента нет описания и имя неоднозначно — кладём в «Прочее» и помечаем в замечаниях.
- Матрица вариантов — строкой `key: val1 | val2 | val3`. Не разворачиваем все 24 комбинации.
- Node ID обязателен. По нему агент при сборке экранов вызывает `use_figma` с конкретным компонентом.
- Описание (`Назначение`) — одной строкой. Если в Figma `description` заполнено — берём из него (с переводом на русский, если в оригинале английский). Если пустое — пишем кратко по имени и помечаем в замечаниях, что описания нет.

### Фаза 3.3: Self-check

После сборки `ds/components.md` пройди по нему сам и сделай honest-review:

- **Сколько компонентов из общего числа имеют пустое `Назначение`?** Если больше 30% — пометь жёлтым в замечаниях, после сканирования нужно доводка руками.
- **Есть ли наборы компонентов с пустым `componentPropertyDefinitions`?** Это либо одиночные компоненты без вариантов (норма), либо забытая настройка (проблема). Помечаем.
- **Есть ли компоненты с подозрительными именами?** `Component 1`, `Frame 234`, `Untitled` — попадают в замечания, проверка через автора ДС.
- **Дубликаты имён.** Если два набора с одним именем — конфликт, агент при сборке выберет первый. Помечаем явно.

## Фаза 5. Контракт ДС в проекте

Цель — закрепить правила работы с ДС на весь дальнейший модуль. После этой фазы при команде «собери экран», «добавь форму», «дорисуй карточку» агент опирается на индекс из `ds/components.md`, использует только Variables из `ds/foundation.md` и не отступает от этих правил без явного «забудь про ДС».

### Фаза 5.1: Создать `ds/CONTRACT.md`

Один файл с правилами проекта. Содержимое адаптируется под источник (своя / корпоративная / дублированная):

```markdown
# DS Contract — правила работы с дизайн-системой в этом проекте

**Источник ДС:** ds_scan (<своя / корпоративная / дублированная Community-библиотека>)
**Figma-файл:** <ссылка на страницу Components>
**Дата сканирования:** <ISO-дата>

## Главное правило

При любом задании, которое касается интерфейса (новый экран, новая форма, новая карточка, доработка существующего), агент работает строго через эту ДС. Сборка идёт композицией существующих компонентов из `ds/components.md` и существующих Variables из `ds/foundation.md`. Прибитые цвета, размеры, радиусы и шрифты в обход Variables запрещены.

## Что значит «через ДС»

1. **Компоненты — только из каталога.** При сборке экрана агент читает `ds/components.md`, выбирает подходящие компоненты по описанию и матрице вариантов, вставляет инстансы через `use_figma` по Node ID. Не рисует похожие с нуля.
2. **Цвета — только через Semantic Variables** из `ds/foundation.md`. Никаких прибитых hex.
3. **Типографика, радиусы, отступы — через Variables.** Из коллекций исходной ДС.
4. **Auto Layout везде.** Любая секция экрана — Auto Layout.

## Что делать, если нужного нет

ВАЖНО — поведение зависит от источника:

### Если источник «своя» или «дублированная Community»

- **Нет компонента под задачу:**
  1. Добавляешь компонент в Figma-файл ДС (страница Components), оформляешь как набор компонентов с матрицей вариантов и описанием.
  2. Перезапускаешь `ds_scan` или дописываешь строчку в `ds/components.md` руками: имя, назначение, варианты, Node ID.
  3. Только после этого используешь на экране.

- **Нет смыслового токена:**
  1. Добавляешь Variable в коллекцию Semantic исходной ДС с привязкой к примитиву.
  2. Дописываешь в `ds/foundation.md`.
  3. Используешь на экране.

### Если источник «корпоративная»

- **Нет компонента под задачу:**
  1. ДС править нельзя — её ведёт другая команда.
  2. Сначала пробуешь собрать паттерн из существующих компонентов.
  3. Если действительно нет — заводишь запись в `ds/patterns.md` с пометкой «локальный паттерн, не часть корпоративной ДС». На экране собираешь композицией существующих, не плодишь свой компонент в файле ДС.
  4. Параллельно — заводишь запрос в команду ДС на добавление. Через месяц `ds_scan` подхватит, локальный паттерн уберётся.

- **Нет смыслового токена:**
  1. Используешь ближайший существующий из Semantic корпоративной ДС.
  2. Если совсем не подходит — отметь в `ds/patterns.md` как ограничение и параллельно заведи запрос в команду ДС.

## Паттерны композиций

`ds/patterns.md` — опциональный файл, который заводится руками. В нём — типовые композиции вашего продукта (форма заказа, карточка товара, шапка таблицы, состояние пустоты, уведомление). Каждый паттерн: имя, из каких компонентов и токенов собирается, Node ID примера в Figma.

Заводится по мере появления типовой раскладки. Окупается на следующем десятке экранов.

## Синхронизация Figma ↔ ds/*.md

`ds/foundation.md` и `ds/components.md` — индекс, по которому агент работает. Расхождение между индексом и Figma ломает следующие сборки.

- ДС в Figma поменялась (своя / дублированная) → перезапускаешь `ds_scan`, перезаписываешь `ds/foundation.md` и `ds/components.md`.
- Корпоративная ДС обновилась — тоже перезапускаешь `ds_scan` периодически (раз в спринт), сравниваешь diff, помечаешь устаревшие записи.
- Не редактируешь индекс руками без сверки с Figma. Если правки руками — пометь дату и причину, иначе следующий `ds_scan` затрёт.

## Что запрещено

- Прибитые цвета, размеры, радиусы, отступы — везде, где есть подходящий Variable.
- Создание нового компонента «на лету» в составе экрана, если его можно собрать из существующих или вынести в ДС (для своей/дублированной).
- Правки в файле корпоративной ДС без согласования с командой ДС.
- Параллельный второй проект с собственной ДС в этой же папке.

## Когда правила можно нарушить

Только при явной команде пользователя «здесь без ДС, ad-hoc» (например, одноразовый рендер для презентации). Тогда отдельный фрейм вне SectionNode, явно помеченный «ad-hoc, не часть продукта».
```

### Фаза 5.2: Указатель в корневом `CLAUDE.md` проекта

Проверь, есть ли в корне проекта `CLAUDE.md`. Если есть — добавь блок-указатель. Если нет — создай минимальный.

**Блок к добавлению:**

```markdown
## Дизайн-система

В проекте проиндексирована дизайн-система через ds_scan. Все правила работы с интерфейсом — в `ds/CONTRACT.md`. При любых задачах, связанных с дизайном экранов, форм, компонентов:

1. Прочитай `ds/CONTRACT.md` целиком.
2. Прочитай `ds/foundation.md` и `ds/components.md` для актуального состояния токенов и UI-кита.
3. Действуй строго по правилам контракта.
```

Любая будущая сессия Claude Code в этой папке начинается с прочтения `CLAUDE.md` → видит указатель → читает `CONTRACT.md` → работает по правилам.

## Правила работы

- **Только верхний уровень страницы Components.** Главное правило. Никакой рекурсии в инстансы. Никакого чтения корня файла. Если получили ссылку не туда — переспросить, не пытаться угадать.
- **Без фазы 5 директива не считается завершённой.** Без контракта индекс остаётся справочником, агент при следующих задачах не обязан его использовать. Контракт — обязательная часть, не опция.
- **Не править исходную ДС.** Если это корпоративная или Community-библиотека — `ds_scan` ничего на канвасе не меняет. Никаких «давай заодно поправлю описания». Описания дополняются в `ds/components.md` руками после сканирования, не в Figma.
- **Описания на русском в выходных файлах.** Даже если в исходной ДС описания на английском — в `ds/components.md` пишем по-русски (переводим существующий description). Это контекст для нашего агента и нашей команды.
- **Матрицы вариантов компактно.** `type: primary | secondary | ghost`, не таблица из 24 строк. Это индекс, не справочник.
- **Self-check обязателен.** Без короткого review после сканирования файл уйдёт в работу с дырами. Self-check показывает, что сканирование стартовая точка, а не финал.
- **Кириллица.** Если локаль ru/ru+en — отдельно пометь в `foundation.md`, поддерживает ли шрифт ДС русский, и нет ли проблем с Щ/Ъ/Ь/Ы. На корпоративных ДС иногда стоит шрифт без кириллицы — это блокер, всплывает в замечаниях.
- **Большие ДС (300+ компонентов) сканируем по разделам.** Если в одной странице Components вообще всё — попроси автора ДС разбить страницу на разделы. Если нельзя — режем сканирование на N прогонов по 50-80 компонентов за раз и склеиваем `components.md`.
- **Обновление индекса.** ДС живёт. Через месяц-два повторить `ds_scan`, перезаписать `ds/components.md`, прогнать diff с предыдущей версией. Это ручная процедура, не часть директивы — но в комментарии указать.
