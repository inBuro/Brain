# Директива: Каталог компонентов в Storybook (storybook)

> **Запуск:** только когда пользователь явно дал эту директиву. Все требования к каталогу описаны здесь — выполняй их, не переспрашивай очевидное (что переносить, какие варианты). Источник состава — `ds/` и собранная база в `src/components/`.

## Задача
Собрать на компонентной базе **аккуратный, качественный каталог** в Storybook: каждый компонент в изоляции со всеми вариантами и состояниями, с живыми переходами между состояниями, плюс песочницы по блокам, где компоненты стоят вместе. Каталог — зеркало системы: и компоненты, и все переменные (смысловые и глобальные). Это инструмент сверки и форма передачи разработке.

## Когда применять
- После `react_base`: в проекте есть `src/components/` (база) и `src/tokens/` (токены), проект поднимается.
- Переиспользуемая: добавились компоненты/варианты — запусти снова, каталог дополнится (Фаза 8). Storybook ставится один раз.

## Предусловие
- Собрана React-база (`directive_react_base`): `src/components/` с компонентами по именам из `ds/components.md`, `src/tokens/` с Primitive+Semantic переменными.
- Node.js 18+ и npm (`node -v && npm -v`).
- Каталог опирается на `ds/components.md` (матрицы вариантов/состояний) и `ds/foundation.md` (все токены).

## Ключевой принцип
Каталог собирается **из настоящих компонентов базы** (`import` из `src/components/`), не перерисовывается заново. Что в базе — то и в каталоге, один в один. Качество каталога = аккуратность: все состояния выложены, переходы анимированы, песочницы показывают компоненты в связке.

## Фаза 0. Контекст
Прочитай `ds/components.md` (состав, матрицы вариантов/состояний) и `ds/foundation.md` (все токены). Сверь со `src/components/index.ts` — что реально собрано в базе. Каталог строишь по пересечению: всё из базы попадает в Storybook.

## Фаза 1. Поставить Storybook
1. Если `.storybook/` уже есть — Storybook стоит, **init не запускай**, переходи к Фазе 2.
2. Иначе: `npx storybook@latest init --yes` (без `--yes` команда задаёт интерактивные вопросы и зависает в безголовом запуске). Init сам определит Vite + React и заведёт `.storybook/`, скрипты `storybook`/`build-storybook` и примеры.
3. Удали авто-сгенерированные примеры (`src/stories/` от init) — каталог собираем свой, мусор не оставляем.
4. Подключи глобальные токены к Storybook: импортируй `src/tokens/*.css` и шрифт в `.storybook/preview.ts(x)`, чтобы компоненты в каталоге выглядели как в приложении.
5. Не удаляй docs-аддон / autodocs из конфига — на нём держатся страницы компонентов с таблицей пропсов-контролов и «Copy code» (см. Фазу 3).

## Фаза 2. Foundation в каталог — все переменные
Заведи раздел `Foundation` (stories, можно MDX-страницами) — это «глобальные и смысловые переменные» наглядно:
- **Primitive (глобальные):** палитра (acc + grays + functional + альфа), шкала размеров шрифта, веса, line-height, tracking, радиусы, отступы, тени — свотчами/строками с именами и значениями.
- **Semantic (смысловые):** все semantic-токены отдельным блоком, с подписью, на какой примитив ссылаются.
- Типографика: каждый TextStyle строкой-примером.
Раздел читает значения из `src/tokens/` (источник — `foundation.md`), не хардкодит.

## Фаза 3. Stories на все компоненты — единый формат (CSF3 + autodocs)
Для **каждого** компонента базы — файл `src/components/<Name>/<Name>.stories.tsx` по шаблону ниже. Формат ОДИНАКОВ для всех — чтобы у каждого компонента была страница с таблицей пропсов-контролов, всеми состояниями и кнопкой **Copy code** (готово к передаче разработчику). Это не импровизация, а обязательный стандарт каталога.

1. **`tags: ['autodocs']` в `meta`** — включает авто-страницу компонента: таблица пропсов (Name / Description / Default / Control), редактируемые контролы и «Show code / Copy code». Без тега этой страницы не будет — тег обязателен у КАЖДОГО компонента.
2. **`argTypes` на каждый проп** с типом контрола: перечисления → `inline-radio`/`select` + `options`; булевы → `boolean`; текст → `text`; объекты/массивы → `object`. Тогда любое состояние крутится живьём прямо в каталоге.
3. **`args`** — осмысленные значения по продукту (реальные данные CryptoVPS, не lorem).
4. **Именованные stories** на каждый значимый вариант/состояние из матрицы `components.md` (Primary/Secondary/Ghost; Default/Error/Disabled; и т.д.).
5. **Story `AllVariants`** с `render` — вся матрица вариантов × состояний рядом на одном холсте (сверка с Figma одним взглядом).
6. **JSDoc** `/** … */` над `meta` и над stories — попадает в описания на autodocs-странице.
7. Импортируй **реальный** компонент из `./`; иконки — Lucide; контент — по продукту.

Канонический шаблон (равняйся на него для всех компонентов):
```tsx
import type { Meta, StoryObj } from '@storybook/react-vite';
import { Settings } from 'lucide-react';
import { Button } from './Button';

const meta = {
  title: 'Components/Button',
  component: Button,
  tags: ['autodocs'],                 // ← страница с таблицей пропсов-контролов + Copy code
  args: { type: 'primary', children: 'Создать сервер' },
  argTypes: {
    type:  { control: 'inline-radio', options: ['primary', 'secondary', 'ghost'] },
    state: { control: 'inline-radio', options: ['default', 'hover', 'pressed', 'disabled'] },
    children: { control: 'text' },
  },
  parameters: { layout: 'centered' },
} satisfies Meta<typeof Button>;
export default meta;
type Story = StoryObj<typeof meta>;

export const Primary: Story   = { args: { type: 'primary' } };
export const Secondary: Story = { args: { type: 'secondary' } };
export const Ghost: Story     = { args: { type: 'ghost' } };
export const Disabled: Story  = { args: { state: 'disabled' } };

/** Матрица Type × State — все варианты рядом. */
export const AllVariants: Story = {
  parameters: { layout: 'padded' },
  render: () => (/* перебор type × state из матрицы components.md, реальные инстансы базы */ null),
};
```

## Фаза 4. Живые переходы
Каталог должен показывать поведение, а не только статичные состояния:
1. Убедись, что у интерактивных компонентов есть CSS-переходы между состояниями (`transition` на цвет фона/обводки/тени при `:hover`/`:active`/`:focus-visible`) — базовые, аккуратные (≈150–200ms ease). Нет — добавь в стили базы (это правка базы, держи её на токенах).
2. В story состояния наведения/нажатия должны отрабатывать нативно (не только как зафиксированный проп), чтобы в каталоге было видно, как состояние плавно меняется.

## Фаза 5. Песочницы по блокам
На каждый логический блок компонентов — story-песочница, где они собраны вместе и реагируют в связке (`title: 'Sandboxes/<Блок>'`):
- **Модальное окно** — Modal с заголовком, Input'ами, Badge и парой Button (secondary + primary) — как ведут себя вместе.
- **Форма** — набор Input/Select/Dropdown + кнопки submit/cancel, с состояниями ошибки.
- **Навигация** — Navbar + Tabs + Breadcrumb + Pagination в одном контексте.
- **Данные/обратная связь** — Table + Toast + Tooltip.
Песочницы собираются из тех же инстансов базы; цель — увидеть компоненты не по одному, а в реальной композиции.

## Фаза 6. Сложные компоненты — опционально глубже
Для сложных компонентов (Table, Select, Dropdown, Toast, Modal) **предложи** пользователю — это по желанию, не делай молча:
> По <Name> можно добавить отдельные кейсы: длинный контент, крайние состояния, микро-анимации появления. Сделать?
Согласен — добавляешь stories на эти кейсы. Нет — пропускаешь, базовый набор уже полный.

## Фаза 7. Поднять и проверить
1. `npm run storybook` (`--ci`, чтобы не открывал браузер сам и не висел на промптах), дождись адреса (обычно `:6006`), открой.
2. Проверь сам: каталог поднялся без ошибок; каждый компонент базы на месте со всеми вариантами; переходы наведения/нажатия отрабатывают; песочницы собираются и реагируют; Foundation показывает все токены.
3. Покажи пользователю адрес и состав. Дальше он проходит каталог рядом с Figma; расхождения называет словами — ты правишь базу или story, не он.

## Фаза 8. Повторный запуск = обновление
Storybook уже стоит: init не трогай. Добавь stories для новых компонентов базы, обнови матрицы у изменившихся, дополни песочницы, если появился новый блок. Дубликаты не плоди, существующее не ломай. В финале перечисли, что добавил/обновил.

- Переходы между состояниями видны живьём хотя бы на Button/Input?
- Песочницы собираются из реальных компонентов и реагируют?

## Грабли (учтено)
- **`storybook init` зависает на промптах** в безголовом запуске → всегда `--yes`. Запуск каталога — `npm run storybook -- --ci` (не открывает браузер, не ждёт ввода).
- **Повторный init ломает конфиг** → если `.storybook/` есть, init не запускаем, только добавляем stories.
- **Авто-примеры от init** (`src/stories/`) оставлять нельзя — выглядит как чужой мусор в каталоге, удаляем.
- **Токены не подключены к preview** → компоненты в каталоге будут без темы/шрифта. Импорт `src/tokens/*.css` в `.storybook/preview` обязателен.
- **Каталог перерисован, а не из базы** → стори должны импортировать реальные компоненты; иначе каталог и приложение разъедутся. Только `import` из `src/components`.
- **Версия Storybook** — ставим `@latest`, без привязки к номеру: init сам подберёт совместимую с Vite/React.
- **Нет `tags: ['autodocs']`** → у компонента не будет страницы с контролами и Copy code, только голый превью-рендер. Тег обязателен КАЖДОМУ — это и есть «оформленный каталог», ради которого урок (то, что отдаётся разработчику).

## Правила
- Каталог — из настоящих компонентов базы, не перерисовка.
- Состав и имена — из `ds/`; ничего не выдумываем сверх базы.
- Только токены в стилях (правки базы тоже на токенах, `CONTRACT`).
- Сложные кейсы/доп. анимации — по желанию пользователя, не молча.
- Designer направляет и смотрит каталог; правки в код вносит агент.
