# Директива: Дизайн-система в React — проект и компонентная база (react_base)

> **Запуск:** только когда пользователь явно дал эту директиву. Ты её выполняешь по шагам, ничего не додумываешь сверх неё. Источник истины — папка `ds/` и Figma; новые токены/компоненты с нуля не изобретаешь.

## Задача
Развернуть React-проект и собрать в нём **зеркальную копию дизайн-системы**: токены из `ds/foundation.md` и компоненты из `ds/components.md`, один в один по именам, по правилам `ds/CONTRACT.md`. На выходе — рабочий проект и тестовая страница, где видно палитру, шкалу текста и все компоненты. Это фундамент: на нём в следующих уроках собираются каталог (Storybook) и экраны.

Это перенос, не пересборка ДС. Имена токенов и компонентов, их варианты и состояния — те же, что в `ds/`. Меняется только носитель: из Figma-описания → в React-код.

## Когда применять
- ДС уже стоит (Figma + папка `ds/` с тремя файлами), среда развёрнута, Figma MCP подключён. Если `ds/` нет — это не этот путь: сначала `ds_scan` / `ds_rebind`.
- Запускается один раз на проект. Повторный запуск — режим **синхронизации** (см. Фазу 6): подтянуть в код то, что поменялось в `ds/`, не сноси проект.

## Предусловие
- В корне проекта есть `ds/foundation.md`, `ds/components.md`, `ds/CONTRACT.md`.
- Установлены Node.js (18+) и npm. Проверь: `node -v && npm -v`. Если их нет — останови и попроси пользователя поставить Node LTS (ничего не выдумывай, sudo не используй).
- Figma MCP подключён (`directive_figma_mcp`) — нужен для сверки точных значений и матриц по месту. Если не подключён, по `ds/` собрать всё равно можно; о недостающей сверке скажи в финале.

## Ключевой принцип
**Имя = якорь.** `surface-default` в Figma → `--surface-default` в коде; `Button` в Figma → `<Button>` в коде с теми же вариантами. На общих именах держится зеркало и вся дальнейшая работа (каталог, экраны, синхронизация). Никаких прибитых значений — только токены, как требует `CONTRACT`.

## Фаза 0. Контекст и тип базы
1. Прочитай `ds/CONTRACT.md` целиком, затем `ds/foundation.md` и `ds/components.md`. Это вход. Дальше работаешь по ним.
2. Определи тип базы — один вопрос пользователю, если не очевидно из `ds/`:
   - **Своя база** (наш случай, CryptoVPS): готовой реализации компонентов нет, собираешь их сам по `components.md`. Идёшь полным путём ниже.
   - **Опенсорсная база** (например, shadcn): у библиотеки есть открытая React-реализация. Тогда: ставишь библиотеку, а её токены/тему **перекрываешь своими** из `foundation.md` (semantic → значения библиотеки), компоненты берёшь из библиотеки и подгоняешь под `CONTRACT`. Переносить вручную почти нечего — пропусти ручную сборку компонентов в Фазе 3, оставь только маппинг токенов и тонкую подгонку.
3. Зафиксируй стек: **Vite + React + TypeScript**, токены — CSS-переменные, стили компонентов — CSS Modules на семантических переменных. (Если в проекте уже выбран другой стек — следуй ему, не ломай.)

## Фаза 1. Каркас проекта
1. Если проекта ещё нет — подними Vite-каркас в текущей папке: `npm create vite@latest . -- --template react-ts`, затем `npm install`. Папку `ds/` и `directives/` не трогай, они остаются рядом.
2. Заведи структуру:
   - `src/tokens/` — `primitives.css`, `semantics.css`, `typography.css`
   - `src/components/` — по компоненту на папку (`Button/Button.tsx` + `Button.module.css` + `index.ts`)
   - `src/components/index.ts` — общий реэкспорт базы
   - `src/Showcase.tsx` — тестовая страница
3. Подключи токены глобально (импорт трёх css в `main.tsx` или `index.css`) до рендера приложения.

## Фаза 2. Токены (из `foundation.md`)
Два слоя, ровно как в Figma. Прибитые значения вне токенов запрещены (`CONTRACT`).
1. **Primitive** → `:root` в `primitives.css`: каждый примитив = CSS-переменная с тем же именем и значением. Цвета (`--accent-500: #2563EB`, вся серая шкала, функциональные + альфа-варианты), размеры шрифта (`--size-*`), веса, line-height, tracking, радиусы (`--radius-*`), отступы (`--space-*`), тени (`--shadow-*`).
2. **Semantic** → `semantics.css`: каждая семантика = переменная, **значением которой служит примитив через `var(...)`** (`--surface-default: var(--gray-00)`). Перенеси все semantic-токены: поверхности, текст, обводки, состояния (bg-success/...). Это двухуровневость Figma в чистом виде: семантика ссылается на примитив.
3. **Типографика** → `typography.css`: на каждый TextStyle из `components.md` (`DS/Heading/lg`, `DS/Body/sm`, `DS/Label/xs` и т.д.) — служебный класс (`.ds-heading-lg { font-size: var(--size-lg); line-height: …; font-weight: var(--weight-medium); }`). Цвет в класс НЕ зашивай — цвет идёт отдельной семантикой (как в Figma TextStyle не несёт цвет).
4. Шрифт Inter подключи (Google Fonts или локально), `font-sans` = Inter.

## Фаза 3. Компоненты (из `components.md`)
Для **каждого** компонента из каталога (Button, Input, Card, Modal, Navbar, IconButton, Badge + расширенные: Select, Dropdown, Tabs, Breadcrumb, Pagination, Tooltip, Toast, Table — все, что есть в `components.md`):
1. React-компонент с **тем же именем**. Матрица вариантов/состояний из каталога → пропсы: `Button` → `type: 'primary'|'secondary'|'ghost'`, `state: 'default'|'hover'|'pressed'|'disabled'|'error'` (по матрице в каталоге; интерактивные состояния — нативно через `:hover`/`:active`/`:disabled`, не только проп). `Input` → `state`, `Badge` → `variant`, и т.д.
2. Стили — **только через семантические переменные** (`background: var(--surface-action-primary)`), радиусы/отступы — через `--radius-*`/`--space-*`. Ни одного прибитого hex/px, где есть токен (`CONTRACT`).
3. Состав каждого компонента бери из раздела «Состав компонента» в `components.md` (паддинги, какой токен где). Сомнения по точному значению — сверь через Figma MCP (`get_variable_defs`/`get_design_context` по Node ID компонента из каталога), но имя и структуру держи по `ds/`.
4. Иконки — заглушки/прозрачные плейсхолдеры (реальные SVG-иконки и визуал — НЕ задача этой директивы). Текст-контент — нейтральные примеры.
5. Реэкспортни всё из `src/components/index.ts`.

## Фаза 4. Тестовая страница (`Showcase.tsx`)
Собери одну страницу, на которой видно, что перенеслось (для быстрой проверки в этом уроке; полный каталог будет в Storybook позже):
- блок палитры — все semantic-цвета свотчами с подписями-именами;
- шкала текста — каждый TextStyle строкой;
- галерея компонентов — каждый компонент во всех вариантах рядом.
Подключи `Showcase` как корневой экран приложения.

## Фаза 5. Поднять и проверить
1. `npm run dev`, дождись локального адреса, открой его.
2. Проверь сам перед отдачей пользователю: страница поднялась без ошибок в консоли; палитра, шкала и компоненты на месте; цвета/радиусы соответствуют `foundation.md`.
3. Покажи пользователю адрес и что на странице. Дальше он сверяет глазами рядом с Figma; правки — его словами, ты вносишь в код, не он.

## Фаза 6. Повторный запуск = синхронизация
Если проект уже собран и директива запущена снова: не пересоздавай каркас. Сравни `ds/` с кодом и перенеси разницу — изменившиеся значения токенов, новые/переименованные компоненты или варианты. Удаляй в коде только то, что удалено в `ds/`. В финале перечисли, что синхронизировал.

## Self-check (перед финалом)
- Все semantic-токены из `foundation.md` есть в `semantics.css` и ссылаются на примитивы (не на прибитые значения)?
- Каждый компонент из `components.md` существует в `src/components/` с тем же именем и полной матрицей вариантов/состояний? Перечисли отсутствующие (для полного переноса список пуст).
- Прогон по коду компонентов: нет ли прибитых hex/px там, где есть токен? Найдёшь — перепиши на переменную.
- `npm run dev` стартует без ошибок, тестовая страница рендерится.

## Финал (вывести пользователю)
```
React-база собрана из ds/.
- Токены: Primitive + Semantic → src/tokens/ (semantic ссылается на primitive)
- Компоненты: <N>/<всего из каталога> → src/components/ (имена и варианты как в Figma)
- Тестовая страница: npm run dev → <адрес>
Не перенеслось / требует внимания: <список или «нет»>
Сверка с Figma: <done / MCP не подключён — сверь значения вручную>
```

## Грабли (учтено)
- **CLI Claude может отсутствовать** в VSCode-расширении — это про `claude mcp add`, не про npm/git. `npm`/`node`/`git` берём через обычный терминал (Bash), они есть.
- **Node/npm не установлены** → останавливаемся и просим пользователя поставить Node LTS. Не ставим сами, sudo не трогаем.
- **Порт занят** — Vite сам возьмёт следующий свободный; адрес читаем из вывода, не хардкодим `5173`.
- **Прибитые значения** — самый частый дефект переноса. Семантика обязана ссылаться на примитив, компонент — на семантику. Прогон Self-check ловит это до отдачи.
- **Цвет в TextStyle** — в Figma TextStyle не несёт цвет; в коде так же: типокласс без цвета, цвет — отдельной семантикой. Иначе текст везде будет одного цвета.
- **Опенсорсная база (shadcn)** — не переноси компоненты руками: ставь библиотеку и перекрывай её тему своими semantic-значениями. Иначе получишь две параллельные базы.

## Правила
- Источник истины — `ds/` + Figma. Имена и состав не меняем, только переносим в код.
- Только токены: ни одного прибитого значения, где есть переменная (`CONTRACT`).
- Figma — read-only: на канвасе ничего не меняем, читаем для сверки.
- Node ID и точные значения не угадываем — берём из `ds/`, при сомнении сверяем через MCP.
- Designer работает на уровне «направляет и смотрит результат», не правит код руками — правки идут через агента.
